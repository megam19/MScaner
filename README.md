## MScaner

Сервис мониторит указанную папку на появление новых `.mxf` файлов, сохраняет информацию о них в SQLite и удаляет устаревшие записи.

### Структура проекта

```
cmd/mscaner/              Composition root: загружает конфиг, собирает зависимости, запускает циклы.
config/                   Загрузка конфигурации из переменных окружения.
internal/
├── domain/file/          Ядро: сущность File, интерфейсы Repository/Scanner, чистая логика Diff.
├── application/          Use-cases: ScanService (сканирование + сохранение), PurgeService (очистка).
├── infrastructure/
│   ├── filesystem/       Реализация Scanner поверх os.ReadDir.
│   └── persistence/
│       └── sqlite/       Реализация Repository поверх modernc.org/sqlite.
└── interfaces/
    └── web/              HTTP-сервер: HTML-страница + JSON API со списком файлов.
```

### Архитектура: Domain-Driven Design

Код разбит на слои так, чтобы **бизнес-правила жили отдельно от технологий**. Если через год вы замените SQLite на Postgres, а HTML на React — ядро проекта не придётся переписывать.

#### Правило зависимостей

Стрелки идут только внутрь, наружу — никогда:

```
      ┌─────────────────────────────────────────────┐
      │                  domain                     │   ← «что» делает система
      │  сущности, value-objects, чистые функции,   │
      │  интерфейсы хранилищ и источников данных    │
      └─────────────────────────────────────────────┘
                         ▲          ▲
                         │          │
      ┌──────────────────┘          └──────────────────┐
      │                                                │
┌─────────────┐                                 ┌──────────────────┐
│ application │   ← «как» это делается, сценарии│  infrastructure  │   ← «чем» делается
│  use-cases  │                                 │ sqlite, fs, и т.д│
└─────────────┘                                 └──────────────────┘
      ▲                                                ▲
      │                                                │
      └──────────────┐                  ┌──────────────┘
                     │                  │
                ┌────────────────────────────┐
                │         interfaces         │   ← «откуда приходят» запросы
                │    web (HTTP), CLI, gRPC   │
                └────────────────────────────┘
                            ▲
                            │
                ┌────────────────────────────┐
                │        cmd/mscaner         │   ← composition root: собирает всё
                └────────────────────────────┘
```

**Ключевое правило:** `domain` НЕ импортирует ничего из `infrastructure`, `interfaces`, `application`. Если вы пишете в `domain/` и руки тянутся к `database/sql` или `net/http` — это сигнал, что код должен жить не здесь.

#### Что в какую папку класть

**`internal/domain/`** — сердце приложения. Сюда попадает:
- **Сущности** (`File`) — структуры данных предметной области и их методы (`IsDeleted`, `IsNew`).
- **Value-объекты** — неизменяемые значения с проверками (например, `FileName`, если бы понадобилась валидация).
- **Интерфейсы хранилищ и источников** (`Repository`, `Scanner`) — домен объявляет, что ему нужно, но не знает, как это устроено.
- **Чистые функции** (`Diff`) — без побочных эффектов, легко тестируются без моков.

Что НЕ класть: SQL, HTTP-хендлеры, парсинг JSON, логирование, работа с диском.

**`internal/application/`** — сценарии использования. Сервис здесь:
- получает данные через доменные интерфейсы,
- зовёт доменную логику,
- сохраняет результат через интерфейсы.

Пример: `ScanService.Run` читает папку, считает `Diff`, вызывает `Save` и `MarkDeleted`. Сервис **не знает**, что под `Repository` стоит SQLite.

Что НЕ класть: CREATE TABLE, ExecContext, json.NewEncoder, os.ReadDir.

**`internal/infrastructure/`** — «всё грязное». Реализации доменных интерфейсов, обёртки над внешними системами:
- `persistence/sqlite/` — SQL-запросы, миграции, нюансы драйвера.
- `filesystem/` — `os.ReadDir`, `filepath`, `io`.

Если появится второй источник данных (S3, сетевая шара с API) — создавайте подпапку рядом: `infrastructure/s3/`. Главное — реализуйте тот же интерфейс из домена (`file.Scanner` / `file.Repository`).

**`internal/interfaces/`** — адаптеры к внешнему миру: то, откуда приходят запросы/команды:
- `web/` — HTTP (HTML и JSON).
- Гипотетически: `cli/`, `grpc/`, `telegram/` — каждый в своей подпапке.

Эти адаптеры **читают и пишут через Repository**, не лезут напрямую в SQLite.

**`config/`** — структура `Config` и функция `Load()`. Всё чтение `os.Getenv`, парсинг, дефолты — здесь. Никто другой не должен читать окружение.

**`cmd/mscaner/`** — единственное место во всём проекте, где встречаются конкретные реализации. Тут `sqlite.Open` + `sqlite.NewFileRepository` превращаются в «просто Repository» и уходят в `application.NewScanService`. Это называется **composition root**.

#### Как добавить новую фичу (пошагово)

Допустим, нужно «отправлять письмо, когда найден новый файл».

1. В `domain/` описать интерфейс: `type Notifier interface { NotifyNew(ctx, File) error }`.
2. В `infrastructure/email/` реализовать SMTP-клиент, удовлетворяющий `Notifier`.
3. В `application/scan_service.go` добавить поле `notifier Notifier` и вызов `s.notifier.NotifyNew(...)` в нужном месте.
4. В `cmd/mscaner/main.go` создать SMTP-клиент и пробросить его в `NewScanService`.
5. При необходимости добавить env-переменные в `config/`.

Обратный порядок (начать с SMTP-кода в сервисе) быстро превращается в кашу и ломает изоляцию слоёв.

### Два режима работы

MScaner рассчитан на два сценария:

1. **Разработка** — правка кода в Docker без установки Go на хост (ниже раздел «Запуск через Docker»).
2. **Продакшен на Windows** — готовый `mscaner.exe` копируется на Windows-сервер и сканирует сетевую SMB-шару по UNC-пути (раздел «Деплой на Windows»). Go на продакшен-хосте не нужен.

### Запуск через Docker

Go на хосте не нужен — весь цикл правки/сборки/запуска происходит в контейнере. При изменении `.go` файлов [air](https://github.com/air-verse/air) автоматически пересобирает и перезапускает сервис.

```bash
# Положите сканируемые .mxf файлы в ./scan-target (или измените volume в docker-compose.yml)
mkdir -p scan-target database

# Запуск
docker compose up --build

# Остановка
docker compose down
```

### Деплой на Windows

Продакшен-сценарий: бинарник крутится на Windows-сервере и сканирует сетевую шару напрямую по UNC-пути (`\\server\share\folder`) — никакого Docker и никаких CIFS-монтирований. Go на Windows-хосте не нужен.

**Шаг 1. Сборка `.exe` через Docker (на машине разработчика):**

```bash
./build-windows.sh
```

Скрипт делает кросс-компил (`GOOS=windows GOARCH=amd64`, `CGO_ENABLED=0`) через `golang:alpine` и кладёт результат в `dist/`:

```
dist/
├── mscaner.exe            # ~12 МБ, одна бинарка, без зависимостей
└── config.env.example
```

**Шаг 2. Деплой на Windows-сервер:**

1. Скопируйте `dist/mscaner.exe` и `config.env.example` в целевую папку (например, `C:\Services\MScaner\`).
2. Переименуйте `config.env.example` → `config.env` и отредактируйте в блокноте. Минимум — задайте `DIR_PATH`:
   ```
   DIR_PATH=\\air-02\imagine_mxf
   ```
   UNC-путь пишется как обычно, без экранирования обратных слэшей.
3. Запустите `mscaner.exe` вручную или зарегистрируйте как службу (удобнее всего через [NSSM](https://nssm.cc/)). Откроется веб-интерфейс на `http://<сервер>:8080`.

**Важно про права доступа к шаре.** На Windows приложение идёт на SMB под учёткой, от имени которой запущен процесс. Убедитесь, что у этой учётки есть права на чтение шары. Если `mscaner.exe` запускается как служба под `LocalSystem` — у неё может не быть сетевого доступа; выберите доменную учётку или локальную с правом на сетевой ресурс.

**Файл конфигурации vs. переменные окружения.** `mscaner.exe` ищет `config.env` сначала в папке с `.exe`, потом в текущей рабочей директории. Реальные переменные окружения имеют приоритет над файлом — можно при желании их выставлять через `SET DIR_PATH=...` в `.bat`-скрипте.

### Конфигурация

Все параметры задаются переменными окружения или через файл `config.env`:

| Переменная              | По умолчанию                  | Описание                                   |
|-------------------------|-------------------------------|--------------------------------------------|
| `DIR_PATH`              | `/scan`                       | Сканируемая директория                     |
| `FILE_EXTENSION`        | `.mxf`                        | Отслеживаемое расширение                   |
| `DATABASE_PATH`         | `./database/sqlite3DB.db`     | Путь к файлу SQLite                        |
| `SCAN_INTERVAL_SEC`     | `120`                         | Интервал сканирования, сек                 |
| `PURGE_INTERVAL_HOURS`  | `24`                          | Интервал очистки, ч                        |
| `RETENTION_DAYS`        | `10`                          | Возраст записей для удаления, дней         |
| `HTTP_ADDR`             | `:8080`                       | Адрес HTTP-сервера                         |
| `HIGHLIGHT_WINDOW_SEC`  | `3600`                        | Окно, в течение которого файл считается «новым» |

### Веб-интерфейс

После запуска откройте [http://localhost:8080](http://localhost:8080) — список файлов с автообновлением каждые 30 секунд. JSON доступен по `GET /api/files`.

Строки подсвечиваются:

- **зелёным** — файлы, добавленные в пределах `HIGHLIGHT_WINDOW_SEC`;
- **красным со зачёркиванием** — файлы, пропавшие из папки (soft-delete: запись в БД сохраняется с `deletedAt`, при повторном появлении файла метка снимается).

Окончательно из БД такие записи убираются при очистке (`RETENTION_DAYS`).
